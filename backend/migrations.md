## Scope
This file owns only:
- Alembic migration rules and backend schema-change workflow.

This file must not contain:
- General database modeling rules, product/domain behavior, or frontend migration concerns.

Source of truth: mandatory

# Migrations

- All application schema changes must go through Alembic.
- Do not modify the database schema manually.
- Do not use `Base.metadata.create_all()` for application schema management.
- Migration files must live in `backend/alembic/versions/`.
- Migration filenames should use a sequential numeric prefix plus a short description.
- Generate migrations from the backend repo with the repo virtualenv:

```bash
APP_ENV=development ENV_FILE=.env.development ./.venv/bin/python -m alembic revision --autogenerate -m "<description>"
```

- Apply migrations from the backend repo with the repo virtualenv:

```bash
APP_ENV=development ENV_FILE=.env.development ./.venv/bin/python -m alembic upgrade head
```

- Review autogenerated migrations before applying them.
- Do not commit an autogenerated migration without checking table names, columns, indexes, constraints, and downgrade behavior.
- Review autogenerated partial indexes carefully and wrap dialect-specific predicates with `sa.text(...)`.
- PostgreSQL enum types created by `sa.Enum(..., name="...")` are not removed by `op.drop_table()`;
  every downgrade that introduces an enum type in upgrade must explicitly drop that type with
  `sa.Enum(name="...").drop(op.get_bind(), checkfirst=True)` or an equivalent reviewed helper.
- `down_revision` must point to the previous migration in the chain unless it is the initial migration.
- Migration commands must not use global `python` or `python3`.
- Production startup runs `alembic upgrade head` before gunicorn.
